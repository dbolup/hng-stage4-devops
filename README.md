# swiftdeploy

A declarative CLI deployment tool that generates Nginx and Docker Compose configurations from a single `manifest.yaml` file, manages the full container lifecycle, and supports canary/stable deployment modes.

---

## How It Works

```
manifest.yaml  ← the only file you edit
      ↓
swiftdeploy init
      ↓
generates → nginx.conf + docker-compose.yml
      ↓
swiftdeploy deploy
      ↓
API container + Nginx container running
      ↓
swiftdeploy promote canary  ←→  swiftdeploy promote stable
```

`manifest.yaml` is the **single source of truth**. Every config file is derived from it. You never hand-write `nginx.conf` or `docker-compose.yml` — the tool does that for you.

## Setup — From Scratch

### Step 1 — Clone the repository

```bash
git clone https://github.com/YOUR_USERNAME/hng-stage4-devops
cd hng-stage4-devops
```

### Step 2 — Install PyYAML

```bash
pip install pyyaml
```

### Step 3 — Make swiftdeploy executable

```bash
chmod +x swiftdeploy
```

### Step 4 — Build the Docker image

The manifest references `swift-deploy-1-node:latest` — build it first:

```bash
docker build -t swift-deploy-1-node:latest .
```

### Step 5 — Review the manifest

Open `manifest.yaml` — this is the only file you ever edit:

```yaml
services:
  image: swift-deploy-1-node:latest
  port: 3000
  mode: stable
  version: "1.0.0"
  restart_policy: unless-stopped

nginx:
  image: nginx:latest
  port: 8080
  proxy_timeout: 30

network:
  name: swiftdeploy-net
  driver_type: bridge

volumes:
  logs: swiftdeploy-logs
```

## Subcommand Walkthrough

### `./swiftdeploy init`

Parses `manifest.yaml` and generates two files from templates:
- `nginx.conf` — from `templates/nginx.conf.j2`
- `docker-compose.yml` — from `templates/docker-compose.yml.j2`

```bash
./swiftdeploy init
```

Expected output:
```
swiftdeploy init
──────────────────────────────────────────────────
  ✔  Generated nginx.conf
  ✔  Generated docker-compose.yml
```

The grader will delete these generated files and re-run `init` to verify they regenerate correctly. If your manifest is valid, they always will.

---

### `./swiftdeploy validate`

Runs 5 pre-flight checks before deploying. Exits non-zero if any check fails.

```bash
./swiftdeploy validate
```

The 5 checks:

| # | Check | What it verifies |
|---|-------|-----------------|
| 1 | manifest.yaml exists and is valid YAML | File exists and parses without errors |
| 2 | All required fields present and non-empty | services.image, services.port, nginx.image, nginx.port, network.name, network.driver_type |
| 3 | Docker image exists locally | Runs `docker image inspect` on the image name |
| 4 | Nginx port is free | Checks port 8080 is not already bound |
| 5 | nginx.conf syntax is valid | Runs `nginx -t` inside a temporary nginx container |

Expected output (all passing):
```
swiftdeploy validate
──────────────────────────────────────────────────
  ✔  manifest.yaml exists and is valid YAML
  ✔  All required fields present and non-empty
  ✔  Docker image 'swift-deploy-1-node:latest' exists locally
  ✔  Nginx port 8080 is free
  ✔  nginx.conf syntax is valid

  Results: 5 passed, 0 failed
  ✔  All checks passed!
```

If a check fails:
```
  ✘  Docker image 'swift-deploy-1-node:latest' not found locally
     → run: docker build -t swift-deploy-1-node:latest .

  Results: 4 passed, 1 failed
```

---

### `./swiftdeploy deploy`

Runs `init`, brings up the full stack, and blocks until health checks pass or 60 seconds elapse.

```bash
./swiftdeploy deploy
```

What it does step by step:
1. Generates `nginx.conf` and `docker-compose.yml`
2. Runs `docker compose up -d --build`
3. Polls `http://localhost:8080/healthz` every 3 seconds
4. Exits 0 when `{"status": "ok"}` is returned
5. Exits 1 if 60 seconds pass with no healthy response

Expected output:
```
swiftdeploy deploy
──────────────────────────────────────────────────
  ✔  Generated nginx.conf
  ✔  Generated docker-compose.yml

  →  Starting containers...
  →  Waiting for health check at http://localhost:8080/healthz ...
  ✔  Stack is healthy! mode=stable uptime=4.2s

NAME      IMAGE                        STATUS
api       swift-deploy-1-node:latest   Up (healthy)
nginx     nginx:latest                 Up
```

---

### `./swiftdeploy promote <canary|stable>`

Switches the deployment mode with a rolling service restart. Only the API container restarts — Nginx stays up throughout, so there is zero downtime.

**Promote to canary:**
```bash
./swiftdeploy promote canary
```

**Promote back to stable:**
```bash
./swiftdeploy promote stable
```

What it does step by step:
1. Updates `mode:` field in `manifest.yaml` in-place
2. Regenerates `docker-compose.yml` with the new `MODE` env var
3. Restarts only the `api` container (`--no-deps`)
4. Polls `/healthz` until the new mode is confirmed
5. Exits 1 if confirmation times out after 60 seconds

Expected output:
```
swiftdeploy promote canary
──────────────────────────────────────────────────
  ✔  Updated manifest.yaml → mode: canary
  ✔  Generated nginx.conf
  ✔  Generated docker-compose.yml
  →  Restarting API container...
  →  Confirming new mode via /healthz ...
  ✔  Mode confirmed: canary
  ✔  Uptime: 3.1s
```

---

### `./swiftdeploy teardown`

Stops and removes all containers, networks, and volumes.

```bash
# Teardown only
./swiftdeploy teardown

# Teardown and delete generated config files
./swiftdeploy teardown --clean
```

Expected output:
```
swiftdeploy teardown
──────────────────────────────────────────────────
  →  Stopping and removing containers, networks, volumes...
  ✔  Stack torn down
  ✔  Deleted nginx.conf
  ✔  Deleted docker-compose.yml
```

## API Endpoints

The API runs on port 3000 internally. All traffic routes through Nginx on port 8080.

### `GET /`

Returns a welcome message with the current mode, version, and timestamp.

```bash
curl http://localhost:8080/
```

```json
{
  "message": "Welcome! Running in stable mode",
  "mode": "stable",
  "version": "1.0.0",
  "timestamp": "2026-04-29T09:53:21+00:00"
}
```

---

### `GET /healthz`

Liveness check. Returns status and process uptime in seconds.

```bash
curl http://localhost:8080/healthz
```

```json
{
  "status": "ok",
  "mode": "stable",
  "version": "1.0.0",
  "uptime_seconds": 42.3
}
```

---

### `POST /chaos` *(canary mode only)*

Simulates degraded behaviour for testing. Only available when `MODE=canary`.

**Slow mode** — sleeps N seconds before every response:
```bash
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "slow", "duration": 3}'
```

**Error mode** — returns HTTP 500 on ~50% of subsequent requests:
```bash
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "error", "rate": 0.5}'
```

**Recover** — cancels any active chaos:
```bash
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "recover"}'
```

In stable mode, the chaos endpoint returns:
```json
{
  "error": "chaos endpoint only available in canary mode"
}
```

## Canary Mode Headers

When running in canary mode, every response includes the header:

```
X-Mode: canary
```

Check it with:
```bash
curl -I http://localhost:8080/
```

```
HTTP/1.1 200 OK
X-Deployed-By: swiftdeploy
X-Mode: canary
Content-Type: application/json
```

## Nginx Behaviour

### Access Log Format

```
$time_iso8601 | $status | ${request_time}s | $upstream_addr | $request
```

Example:
```
2026-04-29T09:53:21+00:00 | 200 | 0.002s | 172.18.0.2:3000 | GET / HTTP/1.1
```

### Custom Headers

Every response includes:
```
X-Deployed-By: swiftdeploy
```

### JSON Error Bodies

Nginx returns structured JSON on gateway errors:

```json
{ "error": "Bad Gateway", "code": 502, "service": "api", "contact": "ops@swiftdeploy.io" }
{ "error": "Service Unavailable", "code": 503, "service": "api", "contact": "ops@swiftdeploy.io" }
{ "error": "Gateway Timeout", "code": 504, "service": "api", "contact": "ops@swiftdeploy.io" }
```

## Security

Containers follow these security practices:

- Run as **non-root user** (UID 1000)
- **Drop ALL Linux capabilities** (`cap_drop: ALL`)
- Re-add only `NET_BIND_SERVICE` where needed
- Service port (3000) is **never exposed** directly — all traffic routes through Nginx
- Images use `python:3.11-slim` — lightweight base under 300MB

## Full Workflow Example

```bash
# 1. Clone and enter repo
git clone https://github.com/YOUR_USERNAME/hng-stage4-devops
cd hng-stage4-devops

# 2. Install dependency
pip install pyyaml

# 3. Make CLI executable

# 14. Tear everything down
./swiftdeploy teardown --clean
```

## Troubleshooting

**Docker image not found:**
```bash
docker build -t swift-deploy-1-node:latest .
```

**Port 8080 already in use:**
```bash
# Find what is using it
sudo lsof -i :8080
# Kill it or change nginx.port in manifest.yaml
```

**Health check times out:**
```bash
# Check container logs
docker compose logs api
docker compose logs nginx
```

**iptables / permission errors:**
```bash
# Make sure Docker has correct permissions
sudo usermod -aG docker $USER
newgrp docker
```

**PyYAML not found:**
```bash
pip install pyyaml --break-system-packages
```
