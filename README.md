# swiftdeploy

A declarative CLI deployment tool that reads a single `manifest.yaml` file and generates all infrastructure configuration automatically — Nginx reverse proxy config, Docker Compose stack, health checks, and rolling deployment mode switching.

> **The manifest is the only file you ever edit. Everything else is generated.**

---

## How It Works

```
manifest.yaml          ← only file you edit
      ↓
./swiftdeploy init
      ↓
generates nginx.conf + docker-compose.yml
      ↓
./swiftdeploy deploy
      ↓
API container + Nginx container running
      ↓
./swiftdeploy promote canary   ←→   ./swiftdeploy promote stable
      ↓
./swiftdeploy teardown
```

The grader will delete generated files and re-run `./swiftdeploy init` to verify they regenerate correctly. If your manifest is valid, they always will.

### 1. Clone the repository
```bash
git clone https://github.com/YOUR_USERNAME/hng-stage4-devops
cd hng-stage4-devops
```

### 2. Install PyYAML
```bash
pip install pyyaml
```

### 3. Make the CLI executable
```bash
chmod +x swiftdeploy
```

### 4. Build the Docker image
The manifest references `swift-deploy-1-node:latest` — build it first:
```bash
docker build -t swift-deploy-1-node:latest .
```

### 5. You are ready
```bash
./swiftdeploy deploy
```

## The Manifest

`manifest.yaml` is the single source of truth. All generated configs are derived from it:

```yaml
services:
  image: swift-deploy-1-node:latest   # Docker image name
  port: 3000                          # Port the API listens on
  mode: stable                        # Deployment mode: stable or canary
  version: "1.0.0"                    # App version injected as env var
  restart_policy: unless-stopped      # Docker restart policy

nginx:
  image: nginx:latest                 # Nginx image
  port: 8080                          # External port Nginx listens on
  proxy_timeout: 30                   # Proxy timeout in seconds

network:
  name: swiftdeploy-net               # Docker network name
  driver_type: bridge                 # Network driver

volumes:
  logs: swiftdeploy-logs              # Named volume for logs
```

## Subcommand Walkthrough

### `./swiftdeploy init`

Parses `manifest.yaml` and generates two files from templates:
- `nginx.conf` from `templates/nginx.conf.j2`
- `docker-compose.yml` from `templates/docker-compose.yml.j2`

```bash
./swiftdeploy init
```

Output:
```
swiftdeploy init
──────────────────────────────────────────────────
  ✔  Generated nginx.conf
  ✔  Generated docker-compose.yml
```

---

### `./swiftdeploy validate`

Runs 5 pre-flight checks before deploying. Exits non-zero if any check fails.

```bash
./swiftdeploy validate
```

| # | Check | What it verifies |
|---|-------|-----------------|
| 1 | manifest.yaml exists and is valid YAML | File exists and parses without errors |
| 2 | All required fields present and non-empty | services.image, services.port, nginx.image, nginx.port, network.name, network.driver_type |
| 3 | Docker image exists locally | Runs `docker image inspect` on the image name |
| 4 | Nginx port is free | Checks the configured port is not already bound |
| 5 | nginx.conf syntax is valid | Runs `nginx -t` inside a temporary nginx container |

Output (all passing):
```
swiftdeploy validate
──────────────────────────────────────────────────
  ✔  manifest.yaml exists and is valid YAML
  ✔  All required fields present and non-empty
  ✔  Docker image 'swift-deploy-1-node:latest' exists locally
  ✔  Nginx port 8080 is free
  ✔  nginx.conf syntax is valid

  Results: 5 passed, 0 failed
  ✔  All checks passed!
```

Output (with a failure):
```
  ✘  Docker image 'swift-deploy-1-node:latest' not found locally
     → run: docker build -t swift-deploy-1-node:latest .

  Results: 4 passed, 1 failed
```

---

### `./swiftdeploy deploy`

Runs `init`, brings up the full stack with Docker Compose, then blocks until health checks pass or 60 seconds elapse.

```bash
./swiftdeploy deploy
```

What happens step by step:
1. Generates `nginx.conf` and `docker-compose.yml`
2. Runs `docker compose up -d --build`
3. Polls `http://localhost:8080/healthz` every 3 seconds
4. Exits 0 when `{"status": "ok"}` is returned
5. Exits 1 and prints logs if 60 seconds pass with no healthy response

Output:
```
swiftdeploy deploy
──────────────────────────────────────────────────
  ✔  Generated nginx.conf
  ✔  Generated docker-compose.yml

  →  Starting containers...
  →  Waiting for health check at http://localhost:8080/healthz ...
  ✔  Stack is healthy! mode=stable uptime=4.2s

NAME      IMAGE                        STATUS
api       swift-deploy-1-node:latest   Up (healthy)
nginx     nginx:latest                 Up
```

---

### `./swiftdeploy promote <canary|stable>`

Switches deployment mode with a rolling restart. Only the API container restarts — Nginx stays up throughout, giving zero downtime.

```bash
# Switch to canary
./swiftdeploy promote canary

# Switch back to stable
./swiftdeploy promote stable
```

What happens step by step:
1. Updates `mode:` field in `manifest.yaml` in-place
2. Regenerates `docker-compose.yml` with the new `MODE` env var
3. Restarts only the `api` container using `--no-deps`
4. Polls `/healthz` until the new mode is confirmed in the response
5. Exits 1 if confirmation does not arrive within 60 seconds

Output:
```
swiftdeploy promote canary
──────────────────────────────────────────────────
  ✔  Updated manifest.yaml → mode: canary
  ✔  Generated nginx.conf
  ✔  Generated docker-compose.yml
  →  Restarting API container...
  →  Confirming new mode via /healthz ...
  ✔  Mode confirmed: canary
  ✔  Uptime: 3.1s
```

---

### `./swiftdeploy teardown`

Stops and removes all containers, networks, and volumes.

```bash
# Stop the stack only
./swiftdeploy teardown

# Stop the stack AND delete generated config files
./swiftdeploy teardown --clean
```

Output:
```
swiftdeploy teardown
──────────────────────────────────────────────────
  →  Stopping and removing containers, networks, volumes...
  ✔  Stack torn down
  ✔  Deleted nginx.conf
  ✔  Deleted docker-compose.yml
```

## API Endpoints

All traffic routes through Nginx on port 8080. The API port (3000) is never exposed directly to the host.

### `GET /`

Returns a welcome message with current mode, version, and UTC timestamp.

```bash
curl http://localhost:8080/
```

```json
{
  "message": "Welcome! Running in stable mode",
  "mode": "stable",
  "version": "1.0.0",
  "timestamp": "2026-05-05T09:25:37+00:00"
}
```

---

### `GET /healthz`

Liveness check. Returns status and process uptime in seconds.

```bash
curl http://localhost:8080/healthz
```

```json
{
  "status": "ok",
  "mode": "stable",
  "version": "1.0.0",
  "uptime_seconds": 42.3
}
```

---

### `POST /chaos` — canary mode only

Simulates degraded behaviour for resilience testing. Returns 403 in stable mode.

**Slow mode** — adds a delay before every response:
```bash
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "slow", "duration": 3}'
```

**Error mode** — returns HTTP 500 on a percentage of requests:
```bash
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "error", "rate": 0.5}'
```

**Recover** — cancels all active chaos:
```bash
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "recover"}'
```

---

## Canary Mode

When running in canary mode, every API response includes an extra header:
```
X-Mode: canary
```

Verify it:
```bash
curl -I http://localhost:8080/
```

```
HTTP/1.1 200 OK
X-Deployed-By: swiftdeploy
X-Mode: canary
Content-Type: application/json
```

## Nginx Behaviour

### Access Log Format

Every request is logged in this custom format:
```
$time_iso8601 | $status | ${request_time}s | $upstream_addr | $request
```

Example lines:
```
2026-05-05T09:25:37+00:00 | 200 | 0.002s | 172.18.0.2:3000 | GET / HTTP/1.1
2026-05-05T09:25:38+00:00 | 200 | 0.001s | 172.18.0.2:3000 | GET /healthz HTTP/1.1
2026-05-05T09:25:39+00:00 | 404 | 0.001s | 172.18.0.2:3000 | GET /missing HTTP/1.1
```

View live logs:
```bash
docker compose exec nginx tail -f /var/log/nginx/access.log
```

### Response Headers

Every response automatically includes:
```
X-Deployed-By: swiftdeploy
```

### JSON Error Bodies

Gateway errors return structured JSON instead of nginx's default HTML pages:

```json
{ "error": "Bad Gateway",         "code": 502, "service": "api", "contact": "ops@swiftdeploy.io" }
{ "error": "Service Unavailable", "code": 503, "service": "api", "contact": "ops@swiftdeploy.io" }
{ "error": "Gateway Timeout",     "code": 504, "service": "api", "contact": "ops@swiftdeploy.io" }
```

## Security Practices

| Practice | Implementation |
|----------|---------------|
| Non-root user | Container runs as UID 1000 |
| Dropped capabilities | `cap_drop: ALL` in compose |
| Port not exposed | API port 3000 never published to host |
| Lightweight image | `python:3.11-slim` base, under 300MB |
| Named volume for logs | Logs isolated in `swiftdeploy-logs` volume |

## Full Workflow — Zero to Running

```bash
# 1. Clone
git clone https://github.com/YOUR_USERNAME/hng-stage4-devops

# 2. Install dependency
pip install pyyaml

# 3. Make CLI executable
chmod +x swiftdeploy

# 14. View access logs
docker compose exec nginx cat /var/log/nginx/access.log

# 15. Tear everything down cleanly
./swiftdeploy teardown --clean
```

## Troubleshooting

**Docker image not found:**
```bash
docker build -t swift-deploy-1-node:latest .
```

**Port 8080 already in use:**
```bash
# Find what is using it
sudo lsof -i :8080
# Or change nginx.port in manifest.yaml to a free port
```

**Health check times out on deploy:**
```bash
# Check container logs for errors
docker compose logs api
docker compose logs nginx
```

**PyYAML not found:**
```bash
pip install pyyaml --break-system-packages
```

**Permission denied running swiftdeploy:**
```bash
chmod +x swiftdeploy
```

**Generated files out of sync after editing manifest:**
```bash
# Always regenerate after any manifest change
./swiftdeploy teardown --clean
./swiftdeploy deploy
```
